# 2단계: 모바일 앱 요청에서도 로그인 유지하기

## 미션 배경

서비스가 모바일 앱으로 확장된다. 사용자는 웹에서만 예약을 관리하지 않고 모바일 앱에서도 로그인하고, 예약을 확인하고, 필요한 작업을 수행하려고 한다.

웹에서 사용한 인증 방식이 모바일 앱 요청에도 그대로 적절한지 검토해야 한다. 브라우저와 모바일 앱은 인증 정보를 저장하고 전달하는 방식이 다를 수 있다.

2단계에서는 모바일 앱 요청에서도 로그인한 사용자를 식별할 수 있게 만든다.

## 목표

- 모바일 앱 요청에서도 로그인한 사용자를 식별한다.
- 인증 정보가 없거나 유효하지 않은 요청을 거부한다.
- 웹 인증 구현과 모바일 인증 구현의 공통점과 차이점을 설명한다.
- 세션과 토큰 중 선택한 방식의 이유를 설명한다.

## 요구사항

- 모바일 로그인 성공 후 모바일 앱이 이후 요청에서 사용할 인증 정보를 받을 수 있어야 한다.
- 인증 정보는 요청마다 서버가 사용자를 식별할 수 있는 형태여야 한다.
- 인증이 필요한 API 호출 시 인증 정보를 함께 전달한다.
- 서버는 전달된 인증 정보를 검증한다.
- 인증 정보가 유효하면 로그인 사용자로 처리한다.
- 인증 정보가 없거나 유효하지 않으면 요청을 거부한다.
- 웹 흐름과 모바일 흐름의 공통점과 차이점을 설명할 수 있어야 한다.
- 가능한 한 중복된 인증 로직을 줄인다.

## 구현 조건

- 세션과 토큰 중 하나를 선택하거나 두 방식을 조합할 수 있다.
- 선택한 방식 외의 다른 후보도 검토한다.
- 인증 정보를 전달하는 위치를 명확히 정한다.
- 컨트롤러는 웹/모바일에 따라 인증 로직이 과도하게 갈라지지 않도록 한다.

## 선택해야 하는 사항

- 모바일 요청의 인증 방식: 세션 유지, 토큰 발급, 웹/모바일 분리 처리 중 무엇을 선택할 것인가?
- 인증 정보 전달 위치: 쿠키, `Authorization` 헤더, 다른 헤더 중 무엇을 선택할 것인가?
- 웹/모바일 흐름 통합 방식: 같은 검증 로직을 재사용할 것인가, 클라이언트별 어댑터를 둘 것인가?
- 만료와 로그아웃 처리: 서버 상태 제거, 토큰 만료, 별도 무효화 저장소 중 무엇을 선택할 것인가?
- 인증 실패 응답: 웹과 모바일에 같은 응답을 줄 것인가, 나눌 것인가?

## 완료 기준

- 모바일 앱 요청에서도 로그인한 사용자를 식별할 수 있다.
- 인증 정보가 없거나 유효하지 않은 모바일 요청을 거부한다.
- 선택한 인증 방식과 다른 후보를 비교해 설명했다.
- 웹 인증 흐름과 모바일 인증 흐름의 공통점과 차이점을 설명했다.
- 완성하지 못했다면 어디서 왜 막혔는지, 어떤 선택 앞에서 망설였는지 정리한다.
