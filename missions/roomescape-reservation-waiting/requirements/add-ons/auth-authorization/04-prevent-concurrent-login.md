# 추가 단계: 동시 로그인 방지

## 미션 배경

서비스 운영팀에서 같은 계정으로 여러 기기에서 동시에 로그인하지 못하게 해 달라는 요청이 도착했다.

처음에는 단순해 보이지만, 이 요구사항은 인증 방식에 따라 구현 난이도가 달라진다. 특히 토큰을 사용하는 경우 서버가 로그인 상태를 기억하지 않는 구조와 충돌할 수 있다.

자율 단계에서는 동시 로그인 방지 요구사항을 가능한 범위에서 구현하거나 설계한다. 완성보다 중요한 것은 어디에서 어려움이 생기는지 이해하는 것이다.

## 목표

- 동시 로그인 방지 요구사항이 인증 방식과 어떻게 연결되는지 이해한다.
- 세션 방식에서 동시 로그인을 제한하는 방법을 검토한다.
- 토큰 방식에서 이미 발급된 토큰을 무효화하는 어려움을 경험한다.
- stateless한 구조가 운영 요구사항과 충돌할 수 있음을 이해한다.

## 요구사항

- 같은 계정으로 여러 기기에서 동시에 로그인하는 상황을 제한한다.
- 새 로그인이 발생했을 때 기존 로그인 상태를 어떻게 처리할지 정한다.
- 기존 로그인을 유지할지, 새 로그인을 우선할지 정책을 정한다.
- 세션 또는 토큰 방식에서 이 요구사항을 구현할 때 어떤 어려움이 있는지 정리한다.

## 구현 조건

- 필수 단계는 아니다.
- 시도한 범위까지만 기록한다.
- 완성하지 못해도 어떤 지점에서 어려웠는지 설명하면 된다.
- 현재 미션에서 선택한 인증 방식을 기준으로 먼저 검토한다.

## 생각해 볼 점

- 서버는 현재 로그인한 사용자의 상태를 알고 있는가?
- 새 로그인이 발생했을 때 기존 로그인 정보를 찾을 수 있는가?
- 이미 발급된 토큰을 무효화할 수 있는가?
- 토큰을 검증할 때마다 서버 저장소를 확인하면 여전히 stateless하다고 볼 수 있는가?
- 운영 요구사항을 만족하기 위해 어떤 상태를 추가로 저장해야 하는가?
