# 사이클 2: 예약 대기 승인

## 한 줄 요약

트랜잭션 경계 규칙을 적용하면서 예약 대기를 승인해 예약으로 전환하고, 예약 대기 순번을 재정렬한다.

## 진행 방식

- 미션은 과제 진행 요구 사항, 기능 요구 사항, 프로그래밍 요구 사항으로 구성된다.
- 기능 구현 전에 기능 목록을 만들고 기능 단위로 커밋한다.
- 기능 요구 사항에 기재되지 않은 내용은 스스로 판단해 구현한다.
- 자동 전환과 수동 승인 중 어떤 방식을 선택할지 직접 결정한다.
- 어디까지를 하나의 트랜잭션으로 묶을지 직접 결정한다.

## 목표

- 토론 활동에서 정한 트랜잭션 경계 규칙을 코드에 적용한다.
- 트랜잭션 경계 결정마다 왜 함께 묶었는지 또는 분리했는지 PR 본문에 남긴다.
- 지금까지 쌓아온 API 설계, 에러 처리, 테스트 전략을 이 기능에 통합 적용한다.

## 기능 요구사항

지금까지는 사용자가 본인의 예약과 대기를 관리할 수 있었지만 대기는 계속 대기로 머무른다. 이번 사이클부터는 대기를 예약으로 전환하는 흐름을 추가하고, 전환 또는 취소 시 슬롯의 대기 순번이 재정렬되도록 한다.

백엔드 API 추가/변경과 사용자가 보는 화면을 함께 진행한다. API에 맞춰 클라이언트 화면을 만들고, 대기 전환과 순번 재정렬이 화면에서 자연스럽게 보이는 것까지 확인한다.

### 1단계: 예약 대기 승인

- 예약 대기를 승인해 예약으로 전환한다.
- 자동 전환 또는 수동 승인 중 직접 선택해 구현한다.
- 자동 전환은 예약 취소 시 대기 1번이 자동으로 예약 전환되는 방식이다.
- 수동 승인은 관리자가 대기를 확인하고 승인 또는 거절하는 방식이다.
- 대기가 예약으로 전환되면 해당 슬롯의 나머지 대기 순번이 재정렬된다.
- 예약이 취소되면 해당 슬롯의 대기 순번이 재정렬된다.
- 요구사항에 명시되지 않은 엣지 케이스를 스스로 식별하고 처리한다.

## 프로그래밍 요구사항

- Java 코드 컨벤션을 지킨다.
- 기본적으로 Woowacourse Java Style Guide를 원칙으로 한다.
- 레벨 1에서 만든 규칙 중 좋은 코드 작성을 위해 의미 있는 규칙을 정리하고 적용한다.
- 이번 사이클에서 구현한 API에 대한 요구사항 테스트를 작성한다.
- 테스트 전략에 맞춰 단위 테스트와 통합 테스트 범위를 결정해 테스트를 작성한다.
- 토론에서 정한 트랜잭션 경계에 맞춰 함께 일어나야 하는 데이터 변경을 묶는다.
- 중간 실패 시 데이터 일관성이 유지되는지 테스트로 확인한다.

## 과제 진행 요구사항

- 구현을 시작하기 전에 기능 요구사항을 분석해 기능 목록을 정리한다.
- `README.md`에 구현할 기능 목록과 직접 결정한 API 명세를 정리한다.
- 커밋 단위는 `README.md`에 정리한 기능 목록 단위로 나눈다.
- AngularJS Git Commit Message Conventions를 참고해 커밋 메시지를 작성한다.
- PR 본문에 트랜잭션 경계 결정, 자동 전환/수동 승인 중 선택한 방식과 각각의 이유를 남긴다.

## 미션 중 기록

- 토론 활동에서 정한 트랜잭션 경계 규칙을 의식하며 코드 작성
- 지금까지의 규칙이 이번 기능에 적용되는 순간과 적용되지 않는 순간
- 스스로 식별한 엣지 케이스와 처리 방향
- 규칙을 적용해서 변경한 코드 1곳 이상
- 테스트 작성이 어려웠던 코드 1곳 이상
- 막힌 순간 1회 이상
