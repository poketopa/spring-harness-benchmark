# 3단계: 인가 - 자기 매장 예약만 관리하기

## 미션 배경

예약 서비스는 여러 매장을 지원한다. 각 매장에는 매니저가 있다. 매니저는 자기 매장의 예약을 관리해야 하지만 다른 매장의 예약을 수정하거나 삭제해서는 안 된다.

로그인한 사용자라고 해서 모든 예약에 접근할 수 있는 것은 아니다. 이제 인증을 넘어 인가가 필요하다.

3단계에서는 매장 매니저가 자기 매장의 예약만 관리하도록 제한한다.

## 목표

- 인증과 인가를 구분한다.
- 로그인한 매니저가 어떤 매장에 속해 있는지 확인한다.
- 접근하려는 예약이 어떤 매장에 속해 있는지 확인한다.
- 매장 매니저가 자기 매장의 예약만 관리할 수 있게 한다.
- 인가 실패와 인증 실패를 구분해 처리한다.

## 요구사항

- 로그인한 사용자가 매장 매니저인지 확인할 수 있어야 한다.
- 로그인한 사용자가 어떤 매장을 관리하는지 확인할 수 있어야 한다.
- 매장 매니저는 자기 매장의 예약만 조회, 변경, 삭제할 수 있다.
- 다른 매장 예약에 접근하려는 요청은 거부한다.
- 로그인하지 않은 요청은 인증 실패로 처리한다.
- 로그인했지만 권한이 없는 요청은 인가 실패로 처리한다.
- 인증 실패와 인가 실패를 같은 문제로 뭉개지 않는다.

## 구현 조건

- 로그인한 사용자가 속한 매장과 접근하려는 예약이 속한 매장을 비교해 요청을 허용하거나 거부한다.
- 인가 판단 로직이 컨트롤러에 과도하게 흩어지지 않도록 한다.

## 선택해야 하는 사항

- 인가 판단 위치: 컨트롤러, 서비스, 도메인 객체 규칙 중 어디에서 판단할 것인가?
- 매니저와 매장의 관계 표현: 매니저가 하나의 매장만 가질 것인가, 여러 매장을 관리할 수 있게 열어둘 것인가?
- 권한 확인 방식: 예약을 조회한 뒤 권한을 비교할 것인가, 처음부터 권한 조건을 포함해 조회할 것인가?
- 인가 실패 응답: 권한 없음으로 응답할 것인가, 리소스를 숨기는 방식으로 응답할 것인가?
- 인증 정보와 도메인 정보의 연결: 로그인 ID로 매니저를 조회할 것인가, 로그인 객체에 매장 정보를 포함할 것인가?

## 완료 기준

- 매장 매니저는 자기 매장의 예약만 관리할 수 있다.
- 다른 매장 예약에 접근하는 요청은 거부된다.
- 인증 실패와 인가 실패가 구분된다.
- 인가 판단 위치와 선택 이유를 설명했다.
- 완성하지 못했다면 어디서 왜 막혔는지, 어떤 선택 앞에서 망설였는지 정리한다.
