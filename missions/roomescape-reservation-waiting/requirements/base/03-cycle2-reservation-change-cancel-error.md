# 사이클 2: 예약 변경, 취소와 에러 처리

## 한 줄 요약

서비스 정책을 적용하면서 잘못된 요청에 사용자가 이해할 수 있는 응답을 주고, 사용자가 본인의 예약을 변경하고 취소할 수 있게 만든다.

## 진행 방식

- 미션은 과제 진행 요구 사항, 기능 요구 사항, 프로그래밍 요구 사항으로 구성된다.
- 기능 구현 전에 기능 목록을 만들고 기능 단위로 커밋한다.
- 기능 요구 사항에 기재되지 않은 내용은 스스로 판단해 구현한다.
- 변경/취소 API 명세, 에러 응답 상태 코드와 본문 형식은 직접 결정한다.

## 목표

- 토론 활동에서 정한 에러 응답 규칙을 코드에 적용한다.
- 잘못된 요청에 500 서버 에러가 노출되지 않게 한다.
- 변경/취소 시 발생하는 에러 케이스를 사용자 관점에서 설계한다.

## 기능 요구사항

지금까지는 정상 흐름인 예약 가능 시간 조회와 예약 생성만 만들었다. 이번 사이클은 잘못된 요청에 대한 응답을 설계하고, 사용자가 본인의 예약을 조회, 변경, 취소할 수 있도록 발전시킨다.

백엔드 API 추가/변경과 사용자가 이해할 수 있는 화면 동작을 함께 진행한다. API와 에러 응답에 맞춰 클라이언트 화면을 만들고, 잘못된 요청 시 사용자가 무엇이 잘못됐는지 알 수 있는 화면까지 확인한다.

### 1단계: 서비스 정책 적용

다음 정책을 만족하지 않는 요청은 거부한다.

- 지나간 날짜와 시간에 대한 예약 생성은 불가능하다.
- 같은 날짜, 시간, 테마에 이미 예약이 있으면 중복 예약을 거부한다.
- 예약이 존재하는 시간을 삭제할 수 없다.
- 유효하지 않은 입력값을 거부한다.
- 유효하지 않은 입력값에는 빈 이름, 잘못된 날짜 형식 등이 포함된다.

### 2단계: 에러 응답 설계

- 서비스 정책 위반, 유효하지 않은 입력, 존재하지 않는 리소스 등에 대해 의도된 에러 응답을 반환한다.
- 500 서버 에러가 사용자에게 노출되지 않도록 한다.
- 에러 응답 본문에 어떤 정보를 담을지 결정한다.
- 브라우저에서 에러 발생 시 사용자에게 의미 있는 메시지가 표시되어야 한다.

### 3단계: 내 예약 조회/변경/취소

- 사용자가 자신의 이름으로 본인의 예약 목록을 조회할 수 있다.
- 사용자가 본인의 예약을 취소할 수 있다.
- 사용자가 본인의 예약 날짜와 시간을 변경할 수 있다.
- 변경/취소 시 발생하는 에러 케이스도 2단계의 규칙에 맞춰 처리한다.
- 에러 케이스에는 이미 지난 예약을 취소하는 경우, 변경하려는 시간이 이미 차 있는 경우 등이 포함된다.

## 프로그래밍 요구사항

- Java 코드 컨벤션을 지킨다.
- 기본적으로 Woowacourse Java Style Guide를 원칙으로 한다.
- 레벨 1에서 만든 규칙 중 좋은 코드 작성을 위해 의미 있는 규칙을 정리하고 적용한다.
- 이번 사이클에서 구현한 API에 대한 요구사항 테스트를 작성한다.
- 예외 처리를 일관되게 한다.

## 과제 진행 요구사항

- 구현을 시작하기 전에 기능 요구사항을 분석해 기능 목록을 정리한다.
- `README.md`에 구현할 기능 목록과 에러 응답 명세를 정리한다.
- 에러 응답 명세에는 직접 결정한 상태 코드와 본문 형식을 포함한다.
- 커밋 단위는 `README.md`에 정리한 기능 목록 단위로 나눈다.
- AngularJS Git Commit Message Conventions를 참고해 커밋 메시지를 작성한다.
